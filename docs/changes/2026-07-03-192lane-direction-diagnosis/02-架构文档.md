# 02-架构文档：96lane vs 192lane 静态诊断

## 一句话判断

`ours_192lane` 的主要风险不是 Python/data 语义错误，而是架构资源压力：AIE kernel 从 192 个增到 384 个，TopPL CU 从 6 个增到 12 个，AIE 输出改成 pktmerge 后端还增加了 packet 解析与宽度转换，`conn.cfg` 又把 stream depth 全部压成 0。继续在这个形态上“微调布局布线”收益很低。

## 关键事实对比

| 项 | ours_96lane | ours_192lane | 影响 |
|---|---:|---:|---|
| TopPL CU | 6 | 12 | PL kernel、AXI master、q0 控制流翻倍 |
| row block | 3 x 1334 rows | 6 x 667 rows | 每个 CU 行数减半，但边界/控制实例翻倍 |
| AIE lanes | 96 | 192 | AIE 并行度翻倍 |
| AIE kernels | 192 | 384 | 每 lane 两核，placement 压力翻倍 |
| AIE lane columns | 24 | 48 | 192 基本铺满更多 AIE 列，布线余量变小 |
| AIE 输入 PLIO | 96 | 192 | 输入端口翻倍 |
| AIE 输出 PLIO | 96 | 48 merged | 端口数降了，但改成 pktstream/pktmerge |
| `conn.cfg` stream_connect | 204 | 264 | 总连接更多 |
| `conn.cfg` depth | 192 条 depth=32，12 条 depth=16 | 264 条全 depth=0 | 192 少了显式 stream slack |
| TopPL 内部 FIFO | lane_stat=2, to/from_aie=128 | lane_stat=1, to_aie=16, from_aie=64 | 为省资源降深，吞吐/死锁余量更小 |
| Makefile impl jobs | 256 | 16 | 编译并行度不同，不是根因但会影响耗时 |

## 关键差异

### 1. AIE placement 压力从“半阵列级”变成“接近全阵列级”

`ours_96lane/aie/Config.h`：`kAieLaneCols = 24`，`kParallelLanesPerCol = 4`，每 lane 两个 kernel，所以 96 lane = 192 kernels。

`ours_192lane/aie/Config.h`：`kAieLaneCols = 48`，192 lane = 384 kernels。按当前放置公式 `lane_tile_col = 1 + lane / 4`，192lane 会使用 AIE columns 1..48；但文件顶部注释仍写 “columns 1..24”，这是 stale comment。

### 2. 192 不是单纯翻倍；它为了少输出 PLIO 引入了 pktmerge

`ours_96lane`：每 lane 一个普通 `output_plio`，`coeff_update` 输出 `output_buffer<float>`。

`ours_192lane`：`coeff_update` 改成 `output_pktstream*`，graph 里每 4 lane 经过 `pktmerge<4>` 合成一个 output PLIO。这样输出 PLIO 从 192 降到 48，但代价是：

- AIE kernel 端手写 packet header/payload；
- PL 端 `TopPL.cpp` 需要 `capture_merged_output_packets()` 解 header、按 way 拆回 4 条 lane stream；
- 每 row 每 lane 从原来的 64 个 64-bit packed words，变成 packet header + 128 个 32-bit payload，再由 PL 端两次读合成一个 64-bit word；
- 端口数降了，但单 merged stream 的瞬时负载、解析逻辑和跨域转换更重。

### 3. PL 侧从 6 个大 worker 变成 12 个 worker，共享 DDR/NoC 压力更高

每个 TopPL 仍负责 16 lane、2000 columns，只是 row block 从 1334 降到 667。总输入数据量没有实质减少，反而多了 6 个 TopPL CU、6 条 stat、6 条 q0 broadcast、更多 kernel control 与 memory traffic 仲裁。

### 4. `conn.cfg` 的 depth=0 是高风险信号

96lane 的 AIE<->PL 主数据流是 depth=32，q0 控制流是 depth=16。192lane 所有 264 条 stream 全部 depth=0。对于已经翻倍的 stream/CU 规模，这意味着没有在 connectivity 层保留显式缓冲 slack；这可能节省资源，但也会降低时序/布线/调度余量。

### 5. 当前本地检查只能证明 192 源码自洽，不能证明可布线

已执行：

```text
cd variants/srad/ours_192lane && python data/test_sim_semantics.py
结果：38 tests OK

python scripts/check_ours_192lane.py
结果：OK
```

也执行了：

```text
cd variants/srad/ours_96lane && python data/test_sim_semantics.py
结果：24 tests, 7 failures
```

这些 96 failures 来自本地 test expectations 与当前 96 文件不同步；用户已说明 96 在板上完美跑通，所以本次不把该测试作为 96 失败证据。

## 排名后的可能根因

1. **最可能：架构规模越过当前 VCK190 实现余量。** 384 AIE kernels + 12 TopPL + 264 stream connections 对 placement/router 太重。
2. **很可能：pktmerge 省端口但没省复杂度。** 输出端口减少，但 packetstream/header/拆包/宽度转换使局部互连和 PL 逻辑更复杂。
3. **很可能：stream depth 全部压成 0 让实现更脆。** 96 有 32/16 的连接缓冲，192 没有。
4. **次要：Makefile 依赖和编译参数不完全一致。** 192 的 `xsa` target 没显式依赖 `$(GRAPH_O) $(KERNEL_XOS)`，但 `make all` 会按顺序调用；这更像工作流风险，不像布局布线根因。

## 建议方向

不要继续把“96 复制翻倍成 192”作为主线。更稳的下一步是做一个最小架构 A/B：

- A：保留 96lane 作为稳定最终基线。
- B：尝试 128/144/160 lane 或减少 TopPL CU，而不是直接 192。
- C：如果必须冲 192，先单独验证两个风险点：`conn.cfg` depth 恢复/分级，以及 pktmerge 是否真的比普通 output PLIO 更可布线。
