# tri_1plio 上板性能优化需求文档

## 目标

在 `variants/stencilflow/tri_1plio` 上排查并优化板上性能，使 256*256*64 大数据量运行耗时：

- 硬目标：反超 `variants/stencilflow/tri` 的 30.9ms 实测基线。
- 期望目标：尽量压到 28ms 以内。
- 正确性目标：输出仍与对应 golden 或可信参考一致。

## 当前现象

- 小数据仿真：256*20 下 `tri_1plio` PL 时间 31.7us，快于 `tri` 的 43.98us。
- 大数据上板：256*256*64 下 `tri_1plio` 33.9ms，慢于 `tri` 的 30.9ms。

## 已确认代码事实

- `tri` 是单输入 PLIO graph：`TopGraph.h` 中 `connect<>(in_plio[0].out[0], core.in)`。
- `tri_1plio` 仍只有 1 个 128-bit PLIO，但在 `TopGraph.h` 中把同一个 PLIO 输出 fanout 到 `core.in[0..2]`。
- `tri` 的 `StencilCoreGraph` 让 lap 输出 4 路 forward，flux1 输出 5 路中间结果给 flux2。
- `tri_1plio` 的 `StencilCoreGraph` 改为 lap 输出 1 个 `sub_pack`，同时送 flux1 与 flux2；flux1 输出压缩后的 `mask_pack`；flux2 还额外读取原始输入窗口。
- 当前 `tri_1plio` 默认 `Makefile`/host/TopPL 为 `64` 行规模；目标 256*256*64 需要 `iter=16384`。

## 待确认问题

- 已确认选择 A：在当前仓库 `variants/stencilflow/tri_1plio` 原地改到 256*256*64，再优化。

## 非目标

- 不优先改 `tri_16plio`、`tri_8plio` 等多 PLIO 设计，除非作为对照实验。
- 不删除生成目录或重做公共仓库结构。
- 不把小数据 emu 波形时间直接当作最终验收指标。
